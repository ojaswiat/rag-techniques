COMP702 M.Sc. Dissertation — Project Briefing

Semantic vs. Statistical vs. Structural: a three-paradigm RAG retrieval benchmark on SEC 10‑K filings

A comparative study of Vector (P1), BM25 (P2), and Summary-Tree (P3) retrieval, run against real financial disclosures, scored by a validated LLM judge, at $0.00 expected spend.

Stage: Planning & Proposal 10-week build, 8 phases 900-run benchmark matrix src/ — not yet started

01 · Research Question

Where does each retrieval paradigm win — and where does it break?

The dissertation asks how three fundamentally different retrieval strategies perform on SEC 10‑K filings — documents that mix dense narrative prose with rigid, table-heavy financial data. The sharpest contrast is semantic vs. statistical (meaning vs. exact words); structural retrieval is included as a third paradigm, deliberately chosen to test whether summarisation can survive table fragmentation.

P1

Semantic (vector)

Retrieval by meaning, via dense embeddings. Expected to shine on synthesis-style questions where wording varies from the source.

P2

Statistical (BM25)

Retrieval by exact-word frequency. Expected to shine on direct fact and figure lookups where phrasing matches the source closely.

P3

Structural (summary tree)

Retrieval by traversing an LLM-generated summary hierarchy. Known risk: summarisation drops cell values and footnotes on dense tables.

02 · Data & Ingestion

Every fact traces back to a stable node_id, never a page number

6–9 SEC 10‑K filings (2–3 companies × 2–3 fiscal years) are parsed with LlamaParse in atomic-table mode. Tables are never bisected — each is preserved as one indivisible TextNode. Because 10‑Ks are natively HTML and page numbers are render-dependent, node_id is the canonical evidence anchor for every metric, citation check, and ground-truth mapping in the entire project; source_page_num is kept only for human readability.

A randomised 20-section manual audit checks for column truncation, table bisection, and markdown rendering errors before any downstream work trusts the node store.

03 · The 140-Query Benchmark

Three disjoint sets, so no data leaks between teaching, testing, and validating the judge

Queries are generated by one model (gpt-oss-120b) and independently verified by another (Qwen3-32b with a search tool) — cross-architecture agreement, not self-confirmation. The verified pool splits into three sets that must never overlap:

PQ · 100

Pipeline Queries

The actual test set. 25 per quadrant. What P1/P2/P3 are benchmarked on. Table: queries.

GQ · 20

Golden Queries

The teaching set. 5 per quadrant, human-labelled with "why this is good". Injected into the Judge's prompt only. Table: golden_queries.

JEQ · 20

Judge Eval Queries

The validation set. 5 per quadrant, hand-scored by the researcher to certify the Judge before it grades anything else. Table: judge_validation.

04 · The Three Pipelines

One shared answerer, three different ways of finding the evidence

All three pipelines answer with the same model at the same settings (Llama 3.3 70B, temperature 0) — so any score difference is attributable to retrieval, not generation. Pipelines see only the query and their own retrieved nodes; never exemplars, ground truth, or answer-location hints.

P1 — Vector RAGretrieval by meaning
Local embeddings (bge-small-en-v1.5) + a local cross-encoder re-ranker (bge-reranker-base) over a FAISS/Chroma index. No metadata pre-filter — an earlier design narrowed search to a target Item, but that was dropped: deciding the section at query time risks leaking the answer location, and BM25 has no equivalent shortcut.
Local CPU · $0Index: FAISS / ChromaDBRe-ranked top-N
P2 — Keyword RAGretrieval by exact words
Pure rank_bm25 (true Okapi BM25) over a custom regex tokenizer that preserves numbers, decimals, %, and currency tokens, strips table-markdown pipes while keeping cell values, and applies no stemming. Deterministic, reproducible, no LLM at retrieval time. LlamaIndex's KeywordTableIndex is explicitly rejected — it calls an LLM and would stop this being a clean statistical baseline.
Local CPU · $0No LLM at query timeFully deterministic
P3 — Structural RAGretrieval by summary-tree traversal
A hierarchical SummaryIndex built once, offline, over parent summaries of child nodes, using llama-3.1-8b-instant (chosen for its high daily request ceiling). Built once, cached to disk, never rebuilt. Documented, not hidden, weakness: summarisation can drop cell values and footnotes, so P3 is expected to underperform on the table-heavy quadrants (Q3/Q4) — that degradation is itself part of what the study measures.
One-time LLM buildExpected weak on Q3/Q4Cached, never rebuilt
Future scope, not built: P4 (brute-force full-context), P5 (naive flat vector, no re-ranker), P6 (BM25→vector hybrid fusion) are documented as reserved extensions, ranked by increasing complexity — kept out of scope to protect the 10-week timeline.

06 · Evaluation & the Judge-Validation Gate

A cheap 60-output gate must pass before the expensive 900-run matrix begins

Precision@K, Recall@K, Evidence Hit Rate — deterministic, code-only, keyed on node_id.

07 · Budget & Guardrails

$0.00 expected spend — the real constraint is Groq's daily token ceiling, not money

Every recurring component sits on a free tier or runs locally on CPU. There is no infrastructure that bills per hour or scales to non-zero. The binding constraint is tokens-per-day (TPD) on the heaviest workloads — answer generation and judging — not request count and not dollars.

Answer generation

~960 calls, ~2M tokens against Llama 3.3 70B's ~100K TPD → roughly 2–3 weeks of intermittent free-tier running. The project's real bottleneck.

Judging

~960 calls; rubric + 5 exemplars prompt-cached (free), so throughput is comfortable across days.

Concurrency

tenacity backoff + jitter on 429s; asyncio.Semaphore capped at ≤5 parallel workers.

Storage

SQLite (WAL mode), 5 tables, results committed per row — a crash resumes instead of re-spending quota.

Worst-case paid fallback (optional, never required): the entire LLM workload is on the order of ~4M tokens, landing around $3–5 at paid rates if the free tier is skipped for speed.

10 · Outsourcing to an Agent Team

If AI agents wrote and ran everything: how much human input is actually required

Most of this project is mechanical enough for a coding-agent team to build and execute end-to-end — ingestion, both index builds, dataset generation, all three pipelines, the 900-run matrix, and the analysis scripts. The touchpoints below are the ones that can't be handed to an agent, because the spec itself designs them as human checks (a judge validated by the same system it's scoring proves nothing). Everything else reduces to "paste keys in .env, watch it run."

Everything else

Ingestion, P3 index build, dataset generation + critique, P1/P2/P3 implementation, the 900-run matrix with resume-on-crash, metrics aggregation, and chart generation — an agent team can run this unattended once the two fixed gates above are cleared per phase.

What "minimal" looks like in practice

Paste keys → agents build through Phase 3 unattended → you spend ~1–2 hrs labelling GQ → agents build Phase 5–6 → you spend ~2–3 hrs hand-scoring JEQ and approve the gate → agents run Phase 7 in the background over ~2–3 weeks (TPD-paced) → you write the dissertation from the agents' Phase 8 output.

Not a technical bottleneck

None of the fixed touchpoints are code review or debugging — they're the specific human judgments the study's validity is built on. Removing them wouldn't just add convenience, it would remove the evidence that the automated judge can be trusted at all.
