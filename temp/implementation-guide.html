Phase 1

Environment, Infrastructure & Cost Guardrails

Week 1

Before touching any data, set up the accounts, the API keys, and the safety rails (rate-limit handling, a "test mode" switch, and the database) everything else will depend on. Nothing here calls an AI model for real — it just builds the plumbing.

15 steps · 10 🤖 AI · 5 🙋 Human

  1. Create a free account at console.groq.com — this is the service that will run every AI model call in the project, for free, on a rate-limited (not billed) tier.🙋 Human required
  2. Generate a Groq API key from the account dashboard and copy it somewhere safe.🙋 Human required
  3. Create a free account at cloud.llamaindex.ai — this is LlamaParse, the tool that turns raw PDF/HTML filings into clean, structured text and tables.🙋 Human required
  4. Generate a LlamaParse API key and copy it somewhere safe.🙋 Human required
  5. Install Python (a recent version) and set up an isolated project environment so this project's libraries don't clash with anything else on the machine.🤖 AI-doable
  6. Create a file named .env in the project folder with two empty placeholder entries: one for the Groq key, one for the LlamaParse key.🤖 AI-doable
  7. Paste the two API keys from steps 2 and 4 into the .env file. This is the only place secrets are typed in by hand.🙋 Human required
  8. Write down the exact version number of every library the project depends on, so a future re-install can't silently break from an update.🤖 AI-doable
  9. Install all the pinned libraries into the project environment.🤖 AI-doable
  10. Design and create a small local database (SQLite) with five separate tables: one for parsed document chunks, one for the test questions, one for the "teaching" questions, one for the "judge validation" questions, and one for every result the project produces.🤖 AI-doable
  11. Add a rule to the database layer that refuses to let the same question exist in more than one of the three question tables.🤖 AI-doable
  12. Build a small wrapper around every AI-model call that automatically waits and retries if the service says "slow down" (a rate-limit error), instead of crashing.🤖 AI-doable
  13. Limit the project to at most 5 AI-model calls happening at the same time, so the free tier is never overwhelmed.🤖 AI-doable
  14. Add a simple on/off switch (a "test mode") to every script that calls an AI model, which — when on — limits it to processing just 3 items instead of the full batch.🤖 AI-doable
  15. Look up Groq's current published rate limits (requests and tokens per day, per model) on their console and write them down with today's date, so later phases know what ceiling they're working under.🤖 AI-doable
Phase 2

Ingestion & Parsing Pipeline

Weeks 1–3

Turn a handful of raw SEC 10-K filings into a clean, well-organised store of text and table chunks. Every later measurement in the project traces back to a chunk created here, so getting this right matters more than any other single phase.

14 steps · 11 🤖 AI · 3 🙋 Human

  1. Decide which 2–3 companies and which 2–3 fiscal years of their annual filings (10-Ks) to use as the study's dataset — this shapes the whole research narrative, so it's a judgment call, not a mechanical one.🙋 Human required
  2. Download the chosen 10-K filings from SEC EDGAR, the SEC's free public filings database.🤖 AI-doable
  3. Configure LlamaParse to run in its "atomic table" mode, so tables come out as clean, intact Markdown instead of being cut into pieces.🤖 AI-doable
  4. Run every filing through LlamaParse and collect the parsed narrative text and table blocks.🤖 AI-doable
  5. Give every parsed chunk a permanent, unique ID (a "node ID") that will be used everywhere downstream instead of a page number.🤖 AI-doable
  6. Tag every chunk with the section heading it belongs to (e.g. "Item 1A: Risk Factors"), so later steps know the surrounding context.🤖 AI-doable
  7. Tag every chunk as either "text" or "table", since the project treats and measures the two differently.🤖 AI-doable
  8. Keep the original page number as a separate, human-readable label only — never something a metric relies on, since PDF page numbers aren't reliable coordinates for HTML filings.🤖 AI-doable
  9. Save every parsed and tagged chunk into the "nodes" table in the database.🤖 AI-doable
  10. Cache every parsed filing to disk so it never has to be re-parsed (and re-billed against the free quota) if the process is re-run.🤖 AI-doable
  11. Randomly pick 20 sections across the parsed filings to manually check for parsing problems.🤖 AI-doable
  12. Actually read through those 20 sampled sections and check whether any table got cut in half, any column got truncated, or the Markdown looks broken.🙋 Human required
  13. Decide, for anything found broken, whether to re-parse it with adjusted settings or drop that filing from the dataset entirely.🙋 Human required
  14. Re-run parsing with any corrected settings and confirm the fixed sections now look clean.🤖 AI-doable
Phase 3

P3 Summary-Index Build (one-time)

Week 3

Build the "summary tree" that the third retrieval strategy (structural search) will search over. This is the only index-building step allowed to use an AI model, and it only ever runs once per filing.

10 steps · 8 🤖 AI · 2 🙋 Human

  1. Group the parsed chunks by section, so each group has a clear set of "child" chunks that will share one summary.🤖 AI-doable
  2. Use a small, fast AI model to write one summary per group of child chunks.🤖 AI-doable
  3. Stack those summaries into a tree structure, so broader summaries sit above more detailed ones.🤖 AI-doable
  4. Save the finished tree to disk/database so it's never rebuilt from scratch again.🤖 AI-doable
  5. Run the whole build once in "test mode" (only 3 items) to catch obvious bugs before spending real quota.🤖 AI-doable
  6. Confirm the test-mode run finished cleanly with no errors before allowing the full build to proceed.🙋 Human required
  7. Release the test-mode limit and build the summary tree for every filing in full.🤖 AI-doable
  8. Log how long the build took and how many AI tokens it used, since this feeds into the project's efficiency comparison later.🤖 AI-doable
  9. Spot-check a few summaries from table-heavy sections to see whether important numbers or footnotes got dropped — this is an expected, documented weakness of this approach, and confirming it happens (rather than assuming) is the point.🙋 Human required
  10. Write a short note recording what was found in the spot-check, for later reference.🤖 AI-doable
Phase 4

Dataset Generation & Adversarial Verification

Weeks 4–5

Produce the 140 test questions the whole project will be graded on — using one AI model to write them and a completely different one to independently double-check them, so no single model's blind spots can bias the dataset.

13 steps · 11 🤖 AI · 2 🙋 Human

  1. Split each filing into per-section pieces small enough to fit inside one AI model call.🤖 AI-doable
  2. For each section, have the "Generator" AI model write candidate questions, their correct answers, and the exact node IDs that prove the answer.🤖 AI-doable
  3. Hide the answer and the citation from a second, different "Critic" AI model — it only gets to see the question text.🤖 AI-doable
  4. Give the Critic model a search tool that can look across every parsed chunk in the filing, not just the section the question came from.🤖 AI-doable
  5. Let the Critic independently search for and cite its own answer to each question.🤖 AI-doable
  6. Compare the Critic's independent answer and citations against the Generator's original ones, using a simple automated check (not another AI call).🤖 AI-doable
  7. Keep questions where the two agree; throw out and regenerate any where they disagree.🤖 AI-doable
  8. Repeat generation and verification until each of the four question types has 35 verified questions (140 total).🤖 AI-doable
  9. Randomly split the 140 verified questions into three separate, non-overlapping groups: 100 for testing the pipelines, 20 for teaching the judge, and 20 for validating the judge.🤖 AI-doable
  10. Run an automated check confirming none of the three groups share a single duplicate question.🤖 AI-doable
  11. For each of the 20 "teaching" questions, read the question and its answer, then write a short note explaining why that answer deserves a good score.🙋 Human required
  12. Give each of those same 20 questions an example score from 1 to 10.🙋 Human required
  13. Save the human notes and scores into the database alongside their questions.🤖 AI-doable
Phase 5

Pipeline Implementation (P1 / P2 / P3)

Weeks 5–6

Build the three retrieval strategies being compared, plus the single shared "answering" step they all use — so any difference in results comes from how each strategy finds evidence, not from how it writes the final answer.

13 steps · 12 🤖 AI · 1 🙋 Human

  1. Build a local "meaning search" index (P1) using a small embedding model that runs on a normal computer, no paid API needed.🤖 AI-doable
  2. Add a second local step that re-ranks P1's top candidates by relevance before they're used.🤖 AI-doable
  3. Build a local "exact keyword" index (P2) using a pure statistical algorithm (no AI model involved at all).🤖 AI-doable
  4. Write a custom text-splitting rule for P2 that keeps numbers, percentages, and currency symbols intact instead of breaking them apart.🤖 AI-doable
  5. Confirm that same text-splitting rule behaves identically whether it's indexing documents or handling a live search query.🤖 AI-doable
  6. Connect P3 to search over the summary tree built back in Phase 3.🤖 AI-doable
  7. Write one shared prompt template that turns "a question plus some retrieved evidence" into a final answer, used identically by all three strategies.🤖 AI-doable
  8. Set the shared answering model to the same, most-deterministic setting for every pipeline, so results aren't randomly different run to run.🤖 AI-doable
  9. Make sure each pipeline only ever receives the question and its own retrieved evidence — never any hint about where the answer actually lives.🤖 AI-doable
  10. Add a check confirming all three pipelines are given roughly the same total amount of evidence text at each retrieval depth, so the comparison stays fair.🤖 AI-doable
  11. Run each of the three pipelines once in test mode to confirm they each produce a real answer.🤖 AI-doable
  12. Read a handful of those test-mode answers and sanity-check that they actually look like reasonable responses to the questions.🙋 Human required
  13. Run an automated check confirming no pipeline is accidentally being given the ground-truth answer or citation.🤖 AI-doable
Phase 6

Judge Build & Validation Gate

Weeks 6–7

Build the automated grader (the "Judge") and then prove, against real human scoring, that it can be trusted before it's allowed to grade the full 900-run study. This gate is the single most important checkpoint in the whole project.

13 steps · 10 🤖 AI · 3 🙋 Human

  1. Write a scoring rubric describing what makes an answer a 1 versus a 10.🤖 AI-doable
  2. Build logic that, before grading any answer, only pulls in the 5 "teaching" examples matching that question's type — never all 20 at once.🤖 AI-doable
  3. Cache the rubric plus those 5 examples so repeated grading calls reuse it for free instead of re-sending it every time.🤖 AI-doable
  4. Build a plain-code check (no AI involved) confirming a pipeline's cited node IDs are actually a subset of the question's real evidence.🤖 AI-doable
  5. Build a word-overlap scoring function as a secondary, lexical measure of answer quality.🤖 AI-doable
  6. Build an exact-match scoring function (with numeric tolerance) for the two question types where a single correct value is expected.🤖 AI-doable
  7. Run the 20 "validation" questions through all three pipelines to produce 60 real answers.🤖 AI-doable
  8. Read all 60 of those answers yourself and score each one from 1 to 10 by hand.🙋 Human required
  9. Run the automated Judge over those same 60 answers, without letting it see the human scores.🤖 AI-doable
  10. Calculate how often the Judge's score agrees with the human score across all 60.🤖 AI-doable
  11. Decide whether that agreement rate is high enough (above 80%) to trust the Judge with the full study.🙋 Human required
  12. If it isn't high enough, decide whether to rewrite the rubric, switch the Judge to a different AI model, or both.🙋 Human required
  13. Re-run the 60-answer validation test after any change, until it clears the bar.🤖 AI-doable
Phase 7

Full Benchmark Execution

Weeks 7–9+ (runs in the background)

Run the full 900-combination test matrix and score every single output. This is the most expensive phase in terms of time and free-tier quota, so it's paced carefully and built to survive interruptions.

11 steps · 9 🤖 AI · 2 🙋 Human

  1. Confirm the Judge cleared its validation gate in Phase 6 before authorising this phase to begin — this is the one decision that unlocks the expensive run.🙋 Human required
  2. Build the full queue of all 900 combinations: 100 questions × 3 pipelines × 3 retrieval depths.🤖 AI-doable
  3. Start working through the queue, with the concurrency limit and retry-on-rate-limit wrapper from Phase 1 active the whole time.🤖 AI-doable
  4. Save each completed run to the database the moment it finishes, rather than waiting to save everything at the end.🤖 AI-doable
  5. If the process crashes or is stopped, automatically pick back up from the last saved run instead of starting over.🤖 AI-doable
  6. Track how many AI tokens have been used today against the free daily ceiling.🤖 AI-doable
  7. Pause and spread the remaining work across additional days whenever the daily ceiling is reached.🤖 AI-doable
  8. Score every completed run using the now-validated Judge.🤖 AI-doable
  9. Run every completed run through the deterministic checks (citation match, word overlap, exact match, precision, recall) too.🤖 AI-doable
  10. Flag any run whose cited evidence doesn't actually support its answer, so it isn't mistaken for a correct result.🤖 AI-doable
  11. Every so often, check in on overall progress and skim a few freshly scored outputs to make sure nothing looks obviously wrong.🙋 Human required
Phase 8

Results Consolidation & Analysis

Weeks 8–10

Turn the raw results table into the tables, charts, and written analysis the dissertation actually needs. The numbers can be produced automatically — the interpretation and academic argument can't.

10 steps · 6 🤖 AI · 4 🙋 Human

  1. Pull every completed and scored run out of the database.🤖 AI-doable
  2. Aggregate the scores by retrieval strategy (P1 vs P2 vs P3), so overall performance can be compared.🤖 AI-doable
  3. Aggregate the scores by retrieval depth (top-3, top-5, top-10 results), to see whether "more evidence" actually helps.🤖 AI-doable
  4. Break the results down by question type, to see exactly where each strategy wins and where it fails.🤖 AI-doable
  5. Generate the comparison tables and charts the dissertation will reference.🤖 AI-doable
  6. Run a check confirming whether the structural strategy (P3) actually did underperform on table-heavy questions, as predicted back in Phase 3.🤖 AI-doable
  7. Review the generated tables and charts and flag anything that looks wrong, surprising, or worth digging into further.🙋 Human required
  8. Write the analysis narrative that explains, in your own words, what the results actually mean.🙋 Human required
  9. Write the discussion and conclusion sections that connect the findings back to the original research question.🙋 Human required
  10. Assemble and format the final dissertation document for submission.🙋 Human required
